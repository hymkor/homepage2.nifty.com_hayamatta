ど うも、社内では、Java or C言語 or シェルスクリプトくらいしか、言語を選択出来ない人が多いなぁ。できれば、もう一つ、スクリプト言語を学んで欲しいところだ。

シェルスクリプトは、操作単位が「ファイル」である故、多数のファイルを扱う場合のが得意だ。反面、一つのファイルをぐちゃぐちゃ触るのは、不可能ではないが、かなり厳しいものがある。こういう場合、AWK , Perl , Ruby のようなスクリプト言語を積極的に使用すべきだ。sed はよく知らんから、パス。AWK は、文法が C 言語に似ているので学びやすいこと、全ての UNIX 系 OS に必ず載っているのが強みだ。ただし、最新の機能を搭載しているコマンド名称が違うので注意が必要である。

Solaris , HP/UX (主に商用UNIX)
nawk (New AWK) , 単なる「awk」とすると Old AWK になってしまうので注意
Linux , BSD
gawk (GNU AWK) 。こちらは単なる「awk」でも GNU AWK のはず。

AWK は実は一つの機能に特化している特殊言語である。入力ファイルを加工して、結果を出力する。要はフィルターである。この利用方法にマッチする使い方では驚くほど、短かいコードで済ますことができる。すばらしい。また、nawk , gawk であれば、BEGIN句を main 関数かわりにすることによって、普通の言語のように使うことも可能である。

難点は正規表現まわりの機能が弱いこと。マッチした文字列の取り出しとかが結構難しい。Perl のように $' とか $1 に相当する変数が使えればよいのだが。gawk では、関数 match と組込変数 RSTART , RLENGTH を使えばよいようだ。match なしのパターンでは使えないのが痛い。nawk は知らん。

とはいえ、ちょっと複雑なシェルスクリプトを書くと、かならず必ずワンライナー的な使い方をするので、少しくらいかじった人も多いだろう。それなら、ついでに極めてしまっても決っして損にはなるまいし、それほど多難な道程にはならないだろう。

(参考) JM: Manpage of GAWK

一 方、Perl では、

  • AWK と違い、作りは素直な汎用言語。
  • ほぼ究極の正規表現・文字列操作が可能になっている
  • 一見するとC言語でないと出来なさそうに思える機能でさえもサポートしていたりする
  • 最近の OS ではほぼ標準装備
  • SE以外の関係者にも、認知されていて、利用するのに社会的障害が少ない

とほぼ完璧である。よくある指摘で、プログラムがゴチャゴチャになりやすいという話があるが、そういうものは心掛けで何とかなると思う。ただ、全く欠点がないわけでもない。どちらかというと、社会的な話で言語に罪はないのだが…

  • 意外とよい初心者向けの安い文献が少ない。

CGI と絡めた話や Perl4 レベルまでの内容の書籍が多く、Perl5 以降の機能やライブラリまわりまで解説した、初心者向けの安いものは意外と無いように思える*11。そのため、ワタシはどうしてもシェルスクリプトの代替としてまで使いこなすことが出来なかったりする。

  • 日本語対応度の弱さ

AWK にも言える話だが、標準の Perl は日本語に対応していない*12。別途 jperl を入れるか、2バイト文字を分割しないよう注意することが必要だ。jcode.pl とか jcode.pm が標準で入っていないことも多いしね。

この辺は実は Ruby の長所の裏返しであったりする。Ruby は解説範囲が広くて分かりやすい書籍が多いし、日本語サポートについては「完璧」だったりする。Perl の代替を目指し、実現しているだけに申し分ない。ということで、ちょっと複雑なことをする時、ついついワタシは Ruby を使ってしまうのだ。後継者問題どうすんだ。そう、Ruby は社会的認知がまだ低いのが問題なんだな*13

で 、以上は UNIX/Linux のお話。Windows の場合、標準では bash も Perl も Ruby も入っていない。では、どうするか。

  • VBScript , JScript (WindowsScriptingHost)を使う
  • ActivePerl , ActiveScriptRubyを導入する。

ActivePerl も ActiveScriptRuby も、msi 形式で、かつ、インストール時に再起動が要らないので、サーバ機にも導入しやすかろう。うまいことやると、Windowsサーバ・UNIX/Linux サーバ両方で動くスクリプトが作成できるし。一方、VBScript , JScript は Windows に特化するが、標準で使えるので、社会的敷居が断然低い。うまいこと使いわけよう。ワタシは ActivePerl と VBScript を両方使って、かなり混乱の極みですが。

これらは全て、Windows の COM が利用できるので「ものすごく強力」であるというのは、その筋の人には常識なのだが、知らん人も多いのよね…。

  • 【Ｃ言語講座】標準ライブラリ関数
    関数リファレンス。これによると、ナロー文字列関数 strXXX に対応して、ワイド版の wcsXXX が存在するようである。

ワイド文字列とナロー文字列への相互変換は、上記ページには見当たらないが、mbstowcs , wcstombs といった関数が各コンパイラのオンラインマニュアルには載っている。

  • HP/UX のテクニカルドキュメント
  • linux.or.jp の JM(Japanese Manpage)
  • Microsoft Developer library : mbstowcs

UTF-8 , SJIS , EUC など、その OS の標準文字コード問題があるため、最初は wcstombs , mbstowcs は非ANSI関数の処理系依存関数かと思いきや、MS のページによれば、しっかりと ANSI 関数であるようである。stdlib.h 内に含まれているようだ。

さて、なぜワイド文字列かというと、激しい文字操作を行う場合、やはり、漢字の 2 バイト目を誤って操作することが絶対にありえるからである。NYAOS〜NYACUS では、結局、個々の操作の場面で一々、ANK文字・漢字の1バイト目・漢字の2バイト目を判断させていたが、どうしても漏れが発生した。こういうのは、やはり、文字列操作ライブラリ内で吸収させるべきなのである。

余談だけど、こういうライブラリ内への仕組みの隠蔽化は常識だと思うんだけど、有名なベンダーでもなおざりにして、開発者に負担をかけているところが多いのが困ったもんだ。wchar_t とは関係ないけど、Oracle の Pro*C/C++ でもvarchar構造体とshort の組み合わせによるベタベタなコーディングを未だにユーザに強いていて、ウンザリ。これじゃあ、いつまでたってもしょうもないオーバーフローやミスの危険から逃れられないよ。だからこそ、Oracle はきちんとした文字列クラスを装備する Java を推すのかもしれないけどね。(なんか、前に言ってたことと、かなり矛盾してるような気もするが)

と偉そうに言ったものの、拙作の NYACUS にしても、バッファオーバーフローこそありえないが、wchar_t 対応については、完全に手遅れだったりする。もっと、早く気がついてたらねぇ。

修 行僧さんの日記にて

  • ワタシが特定のアーキテクチャに依存したくない理由
  • （続）ワタシが特定のアーキテクチャに依存したくない理由

私個人としては、別の側面から激しく同意する話であります。

基本的には、極めてシンプルな話です。ポータビリティー(アーキテクチャへの非依存性/独立性/移植性)や複数の手法の選択性は、コストとのトレードオフ問題にすぎません。複数プラットフォームを想定するのは、何だかんだ言って、見えないコストがかかります。

  1. 依存コード/ライブラリの使用を避けることによる、コーディングの工数アップ
  2. 依存しないコード/複数のプラットフォーム向けコードを書ける優秀なプログラマを使う/確保するための単価アップ

これらのコストが合理的なものかどうかは、結局、そのシステムの寿命によります。新しいアーキテクチャへの対応が必要になるまで、システムが業務の要求に耐えてしまうものであれば、視野に入れて然るべきものです。逆にそこまでもたないシステム、つまり、業務内容が激しく変化し、システム全体の再構築が行われる方が早いようであれば、無駄無駄無駄無駄ぁ〜です。

お客さんの視点から見ると、手段は実はどうでもよくて、「御社の一番得意な技術でよいので、一番安くて確実な方法をとってください」というケースが多いように思われます。この場合、お客さんは技術を選ぶのではなく、業者を選んでいます。結果、一社内でのアーキテクチャ依存はなくならないということになります。難しいものです。

壱プログラム職人の技術の話であれば、「環境に適応/依存し過ぎると恐竜のように滅びやすいので避ける」のは、あたりまえの話なんですけどね。
